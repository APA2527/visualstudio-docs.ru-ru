### <a name="overview"></a>Обзор
Общие элементы управления составляют большую часть пользовательского интерфейса в Visual Studio. Наиболее распространенные элементы управления, используемые в интерфейсе Visual Studio, должны следовать [рекомендациям по взаимодействию Windows Desktop.](/windows/desktop/uxguide/controls) Эта тема специфична для Visual Studio и охватывает особые ситуации или детали, которые дополняют эти руководящие принципы Windows.

#### <a name="visual-style"></a>Визуальный стиль
Первое, что следует учитывать при укладке элементов управления, является ли элементы управления будут использоваться в тематиченном uI. Элементы управления в стандартном ui не являются тематическим uI и должны следовать [обычному стилю Windows Desktop,](/windows/desktop/uxguide/controls)а это означает, что они не повторно шаблонизированы и должны отображаться в их внешнем виде управления по умолчанию.

- **Стандартные (полезные) диалоги:** не тематические. Не перелистывайте шаблон. Используйте основные значения стиля управления по умолчанию.

- **Окна инструментов, редакторы документов, дизайн поверхностей и тематические диалоги:** Используйте специализированный тематический внешний вид с помощью службы цветов.

Избегайте использования текста, который пользователь может захотеть скопировать в буфер обмена. Однако статический текст только для чтения может изменяться, чтобы отразить изменение состояния. Статический текст может быть сам по себе в диалоге без какого-либо сдерживания, когда нет конфликта группировки. Решите, если дополнительные линии коробки действительно необходимы. Примером является отображение пути каталога под разделом, созданным групповой линией, как показано ниже:

### <a name="using-text-in-dialogs"></a>Использование текста в диалогах

Основные рекомендации по тексту в диалогах:

- Этикетки для текстовых коробок, списки коробок и кадры в нетематических диалогах начинаются с глагола, имеют начальный капитал только на первом слове и заканчиваются толстой кишке.

    > Текстовые элементы управления в тематических диалогах следуют [рекомендациям Windows desktop UX](/windows/desktop/uxguide/top-violations) и не принимают конечных знаков препинания, за исключением вопросительных знаков в ссылках справки.

- Этикетки для флажков и кнопок опции начинаются с глагола, начального капитала только на первом слове, и не имеют окончание пунктуации.

- Этикетки для кнопок, меню, пунктов меню и вкладок имеют начальные капиталы на каждое слово (название случае).

- Терминология этикетки должна соответствовать аналогичным ярлыкам в других диалогах.

- Если возможно, попросите писателя/редактора написать или утвердить текст, прежде чем он перейдет к разработчику для реализации.

- Все элементы управления должны иметь метки, за исключением особых обстоятельств, в которых табуирование достаточно.
При необходимости используйте текст помощника.

### <a name="helper-text"></a>Текст помощника

Включено в диалоги, чтобы помочь пользователю понять цель диалога или указать, какие действия следует предпринять. Текст помощника следует использовать только тогда, когда это необходимо, чтобы избежать захламления простых диалогов. Двумя вариациями помощника являются диалог овтерговый и водяной знак.

Следуйте общим местам для помощника текста и быть избирательным в введении новых областей. Общие сценарии для помощника текста являются:

- Помощник текст в диалогах, чтобы дать дополнительное направление о том, как взаимодействовать со сложным диалогом.

- Текст водяного знака в пустых окнах инструментов или диалогов, чтобы объяснить, почему содержимое не видно.

- Панель описания, как в нижней части **окна Свойства**.

- Текст водяного знака в пустом редакторе, чтобы объяснить, какие действия пользователь должен предпринять, чтобы начать работу.

### <a name="dialog-helper-text"></a>Текст подсказки для диалогового окна

Дизайнер пользовательского опыта может помочь определить, когда подходящий текст помощника. Дизайнер может определить, где появляется текст помощника, а также его общее содержание. Помощь пользователя может написать/отсеить фактический текст.

### <a name="watermarks"></a>Водяные знаки

Диалоги извлекают пользу из несколько различных водяных меток руководящих принципов. Потому что диалог может показаться занятым многими элементами uI (метки, текст подсказки, кнопки и другие элементы `ButtonShadow`управления контейнером с текстом), особенно когда те появляются в черном, водяные знаки выделяются лучше в темно-серый (VSColor: ). Обычно водяной знак появляется внутри элемента управления, как окно `Window`списка с белым фоном (VSColor: ).

- Текст отображается в темно-сером цвете (VSColor: `ButtonShadow`). Однако, если водяной знак появляется на среднесеровом `ButtonFace`или других цветах (VSColor: ) фон и `WindowText`есть озабоченность по поводу его читаемости, перейдите с черным текстом (VSColor: ).

- Водяные знаки могут быть по центру или флеш слева. При принятии решений о выравнивании применяйте стандартные правила проектирования. Водяной знак не может быть выбран на заднем плане.

![Пример текста водяного знака](../../extensibility/ux-guidelines/media/WatermarkTextExample.gif)<br />Пример текста водяного знака

### <a name="context-specific-dynamic-text"></a>Контекст-специфический (динамический) текст

Динамический текст может быть использован одним из двух способов в диалоге или безспособном uI: либо в качестве динамического ярлыка, либо в качестве динамического контента.

- Динамическая метка: общее использование динамического текста в описательных панелях, которые предлагают больше информации для выбранного элемента, например в диалоге, который содержит список элементов и свойств для тех элементов, отображаемых в сетке справа. Метка для сетки свойств может быть динамической, так что при выборе элемента слева сетка справа показывает информацию для этого конкретного элемента.

- Динамический текст: может быть полезен в тех случаях, когда вам нужно отображать конкретную информацию, а не общую информацию таким образом, но следует позаботиться о том, чтобы не злоупотреблять.

Если вы хотите, чтобы пользователи имели возможность копировать информацию, динамический текст должен быть в поле текста только для чтения.

## <a name="buttons-and-hyperlinks"></a><a name="BKMK_ButtonsAndHyperlinks"></a>Кнопки и гиперссылки

### <a name="overview"></a>Обзор
Кнопки и элементы управления ссылками (гиперссылки) должны следовать [основным указаниям Windows Desktop по гиперссылкам](/windows/desktop/uxguide/ctrl-links) для использования, формулировок, размеров и интервалов.

### <a name="choosing-between-buttons-and-links"></a>Выбор между кнопками и ссылками
Традиционно, кнопки были использованы для действий и гиперссылки были зарезервированы для навигации. Кнопки могут быть использованы во всех случаях, но роль ссылок была расширена в Visual Studio, так что кнопки и ссылки являются более взаимозаменяемыми в некоторых условиях.

При использовании командных кнопок:

- Основные команды

- Отображение окон, используемых для сбора входных или выбор, даже если они являются вторичными командами

- Разрушительные или необратимые действия

- Кнопки обязательства в мастерах и потоках страниц

Избегайте кнопок команд в окнах инструментов или если для метки требуется более двух слов. Ссылки могут иметь более длинные метки.

 Когда использовать ссылки:

- Навигация на другое окно, документ или веб-страницу

- Ситуации, требующие более длинного ярлыка или короткого предложения для описания цели действия

- Тесные пространства, где кнопка будет подавлять uI, при условии, что действие не является разрушительным или необратимым

- Де-подчеркивание вторичных команд в ситуациях, когда есть много команд

### <a name="special-buttons"></a>Специальные кнопки

#### <a name="browse-buttons"></a>Просмотреть... Кнопки
**«Просмотрите...»** кнопки используются в сетках, диалогах, окнах инструментов и других бесрежимных элементах uI. Они отображают сборщик, который помогает пользователю в заполнении значения в элемент управления. Есть два варианта этой кнопки, длинные и короткие.

![Длинная кнопка «Просмотр...»](../../extensibility/ux-guidelines/media/070703-04_browselong.gif "070703-04_BrowseLong")<br />Длинная кнопка «Просмотр...»

![Эллипсис только короткие кнопки](../../extensibility/ux-guidelines/media/070703-05_browseshort.gif "070703-05_BrowseShort")<br />Эллипсис только короткие кнопки

Когда использовать эллипсис только короткая кнопка:

- Если есть более чем одна длинная **кнопка «Просмотр...»** в диалоге, например, когда несколько полей позволяют просматривать. Используйте короткую кнопку **для** каждого из них, чтобы избежать запутанных ключей доступа, созданных этой ситуацией** (&Browse** и **B&строки** в том же диалоге).

- В плотном диалоге, или когда нет разумного места, чтобы положить длинную кнопку.

- Если кнопка появится в управлении сеткой.

Руководящие принципы использования кнопки:

- Не используйте ключ доступа. Чтобы получить к нему доступ с помощью клавиатуры, пользователь должен вкладки из соседнего управления. Убедитесь, что порядок вкладки таков, что любая кнопка просмотра падает сразу после поля, которое она заполняет. Никогда не используйте подчеркивание ниже первого периода.

- Установите свойство Microsoft Active Accessibility (MSAA) **Name** для **просмотра...** (включая эллипсис), чтобы читатели экрана читали его как "Browse", а не "точка-точка" или "период-период-период". Для управляемых элементов управления это означает настройку свойства **AccessibleName.**

- Никогда не используйте кнопку эллипсиса для чего-либо, кроме действия просмотра. **[...]** Например, если вам нужна кнопка **«Новая...»,** но у вас нет достаточно места для текста, то диалог должен быть переработан.

### <a name="hyperlinks"></a>Гиперссылки
Гиперссылки хорошо подходят для действий на основе навигации, таких как открытие темы справки, модальный диалог или мастер. Если гиперссылка используется для команды, она всегда должна отображать видимое и заметное изменение в uI. Как правило, действия, совершающие действия (например, Сохранить, Отменить и удалить), лучше передавать с помощью кнопки.

#### <a name="writing-style"></a>Стиль письма
Следуйте [руководству Windows Desktop для текста пользовательского интерфейса.](/windows/desktop/uxguide/text-ui) Не используйте фразу "Узнать больше", "Расскажи мне больше" или "Получить помощь в этом". Вместо этого фраза Help ссылка текста с точки зрения основного вопроса ответил на содержание справки. Например,**«Как добавить сервер в Server Explorer?»**

#### <a name="visual-style"></a>Визуальный стиль

- Гиперссылки всегда должны использовать [службу VSColor.](../../extensibility/ux-guidelines/colors-and-styling-for-visual-studio.md#BKMK_TheVSColorService) Если гиперссылка не стилизована правильно, она мигает красным цветом, когда активна или показывает другой цвет после посещения.

- Не включайте подчеркивания в состоянии покоя управления, если ссылка не является фрагментом предложения в полном предложении, как в водяном знаке.

- Подчеркивания не должны отображаться на нависке. Вместо этого, обратная связь с пользователем, что ссылка активна, является небольшим изменением цвета и соответствующим курсором ссылки.

## <a name="tree-views"></a><a name="BKMK_TreeViews"></a>Виды деревьев

Представления деревьев предоставляют возможность организации сложных списков в родительские группы. Пользователь может расширить или свернуть родительские группы, чтобы выявить или скрыть основные элементы ребенка. Каждый элемент в представлении дерева может быть выбран для обеспечения дальнейших действий.

### <a name="tree-view-visual-style"></a><a name="BKMK_TreeViewVisualStyle"></a>Визуальный стиль просмотра дерева

#### <a name="expanders"></a>Расширители
Элементы управления представления деревьем должны соответствовать дизайну расширителя, используемому Windows и Visual Studio. Каждый узла использует элемент управления расширителем для выявления или сокрытия основных элементов. Использование элемента управления расширяющих устройства обеспечивает согласованность для пользователей, которые могут столкнуться с различными представлениями деревьев в Windows и Visual Studio.

![Правильно: правильный стиль узла представления дерева с помощью управления расширителем](../../extensibility/ux-guidelines/media/070705-1_treeviewcorrect.png "070705-1_TreeViewCorrect")<br />Правильно: правильный стиль узла представления дерева с помощью управления расширителем

![Неправильно: неправильный стиль узла представления дерева](../../extensibility/ux-guidelines/media/070705-2_treeviewincorrect1.png "070705-2_TreeViewIncorrect1")<br />Неправильно: неправильный стиль узла представления дерева

#### <a name="selection"></a>Выбор
При выборе узла в представлении дерева изюминка должна расширяться до полной ширины элемента управления представления дерева. Это помогает пользователям четко определить, какой элемент они выбрали. Цвета выбора должны отражать текущую тему Visual Studio.

![Правильно: изюминка выбранного узла соответствует всей ширине управления видом дерева.](../../extensibility/ux-guidelines/media/070705-1_treeviewcorrect.png "070705-1_TreeViewCorrect")<br />Правильно: изюминка выбранного узла соответствует всей ширине управления видом дерева.

![Неправильно: изюминка выбранного узла не соответствует всей ширине управления видом дерева.](../../extensibility/ux-guidelines/media/070705-3_treeviewincorrect2.png "070705-3_TreeViewIncorrect2")<br />Неправильно: изюминка выбранного узла не соответствует всей ширине управления видом дерева.

#### <a name="icons"></a>Значки
Значки следует использовать только в элементах управления представления деревьев, если они помогают визуально идентифицировать различия между элементами. Как правило, иконы следует использовать только в неоднородных списках, в которых иконки несут информацию для дифференциации типов элементов. В однородном списке, используя значки часто можно рассматривать как шум и следует избегать. В этом случае значок группы (родитель) может передавать тип элементов внутри него. Исключением из этого правила будет, если значок является динамическим и используется для обозначения состояния.

#### <a name="scroll-bars"></a>Полосы прокрутки
Прокрутка баров всегда должна быть скрыта, если содержимое вписывается в управление представления дерева. Допустимо, чтобы прокрутки были скрыты или полупрозрачными в прокрутке и появляются, когда окно, содержащее вид дерева, фокусировалось или нависло над самим видом дерева.

![Отображаются как вертикальные, так и горизонтальные полосы прокрутки, поскольку содержимое превысило пределы управления представлением дерева.](../../extensibility/ux-guidelines/media/070705-4_scrollbars.png "070705-4_Scrollbars")<br />Отображаются как вертикальные, так и горизонтальные полосы прокрутки, поскольку содержимое превысило пределы управления представлением дерева.

### <a name="tree-view-interactions"></a><a name="BKMK_TreeViewInteractions"></a>Взаимодействия представления дерева

#### <a name="context-menus"></a>Контекстные меню
Узла представления дерева может выявить параметры подменю в контекстном меню. Как правило, это происходит, когда пользователь имеет право нажать элемент или нажал клавишу меню на клавиатуре Windows с выбранным элементом. Важно, чтобы узла фокусистом и выбором. Это помогает пользователю определить, к какому элементу принадлежит подменю.

![Элемент, генерируемый в контекстном меню, получает фокус, чтобы уведомить пользователя, какой элемент был выбран.](../../extensibility/ux-guidelines/media/070705-5_contextmenu.png "070705-5_ContextMenu")<br />Элемент, генерируемый в контекстном меню, получает фокус, чтобы уведомить пользователя, какой элемент был выбран.

#### <a name="keyboard"></a>Клавиатура
Представление дерева должно обеспечить возможность выбора элементов и расширения/коллапса узлов с помощью клавиатуры. Это гарантирует, что навигация соответствует нашим требованиям доступности.

##### <a name="tree-view-control"></a>Управление представлением дерева
Элементы управления деревьями Visual Studio должны следовать общей навигации клавиатуры:

- **Вверх Стрелка:** Выберите элементы, перемещая вверх по дереву

- **Вниз Стрелка:** Выберите элементы, перемещаясь вниз по дереву

- **Правая стрелка:** Расширить узла на дереве

- **Левая стрелка:** Свернуть узла на дереве

- **Введите ключ:** Инициировать, загружать, выполнять выбранный элемент

##### <a name="trid-tree-view-and-grid-view"></a>Trid (вид дерева и вид сетки)
Контроль засушливого из трях — это сложный элемент управления, содержащий вид дерева в сетке. Расширение, рушится и навигация по дереву должны уважать те же команды клавиатуры, что и вид дерева, со следующими дополнениями:

- **Правая стрелка:** Расширьте узла. После расширения узла он должен продолжить навигацию к ближайшей колонне справа. Навигация должна прекратиться в конце строки.

- **Вкладка:** Переходит к ближайшей ячейке справа.  В конце строки навигация продолжается до следующего ряда.

- **Сдвиг и вкладка:** Переходит к ближайшей ячейке слева.  В начале строки навигация продолжается до самой правой ячейки в предыдущем ряду.

![Контроль засушливых элементов в Visual Studio](../../extensibility/ux-guidelines/media/070705-6_trid.png "070705-6_Trid")<br />Контроль засушливых элементов в Visual Studio
