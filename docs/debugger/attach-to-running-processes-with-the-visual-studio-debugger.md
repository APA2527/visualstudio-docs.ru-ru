# <a name="attach-to-running-processes-with-the-visual-studio-debugger"></a>Присоединение к выполняемым процессам с использованием отладчика Visual Studio
Отладчик Visual Studio можно подключить к запущенному процессу на локальном или удаленном компьютере. После запуска процесса выберите **Debug** > **Attach to Process** или нажмите **Ctrl**+**Alt**+**P** в Visual Studio и используйте **прикрепить к диалогу Process для** присоединения отладчика к процессу.

Вы можете использовать **Присоединение к процессу** для отладки запущенных приложений на локальных или удаленных компьютерах, отладки нескольких процессов одновременно, отладки приложений, которые не были созданы в Visual Studio, или отладки любого приложения, которое вы не запускали с Visual Studio с прикрепленным отладчиком. Например, если вы работаете с приложением без отладчика и попадаете в исключение, можно прикрепить отладчика к процессу запуска приложения и начать отладку.

> [!TIP]
> Не знаете, следует ли использовать **Присоединение для обработки** для вашего сценария отладки? Посмотреть [сценарии отладки общего](#BKMK_Scenarios)пользования.

## <a name="attach-to-a-running-process-on-your-local-machine"></a><a name="BKMK_Attach_to_a_running_process"></a>Прикрепите к процессу выполнения на локальной машине

Чтобы быстро прикрепить к процессу, который вы прикрепили к ранее, [см. Reattach к процессу.](#BKMK_reattach)

Чтобы отладить процесс на удаленном компьютере, см [Прикрепите процесс на удаленном компьютере.](#BKMK_Attach_to_a_process_on_a_remote_computer)

::: moniker range=">= vs-2019"
Чтобы отладить процесс .NET Core в контейнере Linux Docker, [см.](#BKMK_Linux_Docker_Attach)
::: moniker-end

**Чтобы прикрепить к процессу на локальном компьютере:**

1. В Visual Studio выберите **Debug** > **Attach to Process** (или нажмите **Ctrl**+**Alt**+**P),** чтобы открыть поле для диалога **Процесса.**

   **Тип соединения** должен быть установлен на **значение по умолчанию.** **Целью подключения** должно быть локальное название машины.

   ![DBG_Basics_Attach_To_Process](../debugger/media/DBG_Basics_Attach_To_Process.png "DBG_Basics_Attach_To_Process")

2. В **списке доступных процессов** найдите и выберите процесс или процессы, к которых вы хотите прикрепить.

   - Чтобы быстро выбрать процесс, введите его имя или первую букву в поле **процессов фильтра.**

   - Если вы не знаете имя процесса, просмотрите список или просмотрите [сценарии отладки](#BKMK_Scenarios) для некоторых общих имен процессов.

   >[!TIP]
   >Процессы могут запускаться и останавливаться в фоновом режиме, пока находится окно диалога **Attach to Process,** поэтому список запущенных процессов не всегда может быть актуальным. Вы можете выбрать **Обновление** в любое время, чтобы увидеть текущий список.

3. В **поле Присоединения** убедитесь, что тип кода, который вы планируете отладить, указан. **Автоматическое** настройки по умолчанию работает для большинства типов приложений.

   Для выбора типов кода вручную:
   1. Нажмите **Выберите**.
   1. В диалоговом окне **типа Select Code** выберите **Debug эти типы кода.**
   1. Выберите типы кода, которые вы хотите отладить.
   1. Нажмите кнопку **ОК**.

4. Выберите **Подключить**.

>[!NOTE]
>Вы можете быть прикреплены к нескольким приложениям для отладки, но только одно приложение активится в отладчике за один раз. Вы можете установить активное приложение в панели инструментов Visual Studio **Debug Location** или в окне **процессов.**

Удаленный отладчик *(msvsmon.exe)* должен работать на удаленном компьютере. Для получения дополнительной [информации см.](../debugger/remote-debugging.md)

Возможно, вам придется [открыть порт UDP 3702](../debugger/remote-debugger-port-assignments.md) на сервере, чтобы обнаружить удаленные устройства. Выберите нужный компьютер или устройство, а затем нажмите **Выберите**.

В некоторых случаях при отладке сеанса удаленного рабочего стола (Terminal Services) в списке **доступных процессов** не будут отображаться все доступные процессы. Если вы работаете visual Studio в качестве пользователя с ограниченной учетной записью пользователя, список **доступных процессов** не будет отображать процессы, запущенные в сеансе 0. Сессия 0 используется для служб и других серверных процессов, включая *w3wp.exe*. Можно устранить эту проблему, запустив [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] под учетной записью администратора или запустив [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] с консоли сервера вместо сеанса служб терминалов.

Если ни один из этих обходных путей использовать невозможно, третий вариант — присоединение к процессу путем запуска `vsjitdebugger.exe -p <ProcessId>` из командной строки Windows. Вы можете определить идентификатор процесса с помощью *tlist.exe*. Чтобы получить файл *tlist.exe*, скачайте и установите средства отладки для Windows, которые доступны на странице [скачиваемых файлов WDK и WinDbg](/windows-hardware/drivers/download-the-wdk).

## <a name="attach-to-a-process-running-on-a-linux-docker-container"></a><a name="BKMK_Linux_Docker_Attach"></a>Прикрепите к процессу, работая на контейнере Linux Docker

Отладка Visual Studio можно прикрепить к процессу, работая в контейнере Linux .NET Core Docker на локальной или удаленной машине, используя **поле** для обработки диалогов.

> [!IMPORTANT]
> Чтобы использовать эту функцию, необходимо установить рабочую нагрузку по разработке кросс-платформы .NET Core Икс и получить локальный доступ к исходному коду.

**Чтобы прикрепить к процессу выполнения в контейнере Linux Docker:**

1. В Visual Studio выберите **Debug > приложите к процессу (CTRL-ALT-P),** чтобы открыть поле для диалога **«Присоединение к процессу».**

![Прикрепите к меню процесса](../debugger/media/attach-process-menu.png "Attach_To_Process_Menu")

2. Установите **тип подключения** к **Docker (Linux Container).**
3. Выберите **Найти...,** чтобы установить **цель подключения** через поле диалога **Select Docker Container.**

    Можно отладить процесс контейнера Docker как локально, так и удаленно.
    
    **Для отладки процесса контейнера Docker локально:**
    1. Установить **Docker CLI принимающей** **местной машины**.
    1. Выберите бегущий контейнер для присоединения к из списка и нажмите **OK.**
    
    ![Выберите меню контейнеров докера](../debugger/media/select-docker-container.png "Select_Docker_Container_Menu")
 
    **B. Для отладки процесса контейнера Docker удаленно:**
    
    > [!NOTE] 
    > Существует два варианта удаленного подключения к процессу выполнения в контейнере Docker. Первый вариант, чтобы использовать SSH, идеально подходит, если у вас нет инструментов Docker, установленных на локальной машине.  Если у вас есть инструменты Docker, установленные локально, и у вас есть докер daemon, который настроен на прием удаленных запросов, попробуйте второй вариант, используя Docker daemon.

    1. ***Для подключения к удаленной машине через SSH:***
        1. Выберите **Добавить...,** чтобы подключиться к удаленной системе.<br/>
        ![Подключение к удаленной системе](../debugger/media/connect-remote-system.png "Подключение к удаленной системе")
        1. Выберите бегущий контейнер для присоединения к после подключения к SSH или daemon успешно и ударил **OK**.

    
    1. ***Установить цель в удаленный контейнер, работающий в процессе с помощью [Docker daemon](https://docs.docker.com/engine/reference/commandline/dockerd/)***
        1. Укажите адрес daemon (т.е. через TCP, IP и т.д.) под **хостом Docker (необязательно)** и нажмите на ссылку обновления.
        1. Выберите бегущий контейнер для крепления после успешного подключения к daemon и нажмите **OK.**

4. Выберите соответствующий контейнерный процесс из списка **доступных процессов** и выберите **Прикрепить,** чтобы начать отладку процесса контейнера C' в Visual Studio!

    ![Завершено Докер Прикрепить Меню](../debugger/media/docker-attach-complete.png "Завершено Linux Docker Прикрепить меню")

## <a name="reattach-to-a-process"></a><a name="BKMK_reattach"></a>Прикрепить к процессу

Вы можете быстро прикрепить к процессам, которые вы были ранее привязаны к, выбрав **Debug** > **Reattach к процессу** (**Shift**+**Alt**+**P**). При выборе этой команды отладчик немедленно попытается прикрепить к последним процессам, к которым вы прилагаете, сначала пытаясь сопоставить предыдущий идентификатор процесса и если это не удается, сопоставляя с предыдущим именем процесса. Если совпадения не найдены, или если несколько процессов имеют одно и то же имя, откроется диалоговое поле **Attach to Process,** чтобы вы могли выбрать правильный процесс.

> [!NOTE]
> Команда **Reattach to Process** доступна начиная с Visual Studio 2017.

## <a name="common-debugging-scenarios"></a><a name="BKMK_Scenarios"></a>Общие сценарии отладки

Чтобы помочь вам определить, следует ли использовать **Присоединение к процессу** и к какому процессу прикрепить, в следующей таблице показаны несколько распространенных сценариев отладки со ссылками на дополнительные инструкции, где это возможно. (Список не является исчерпывающим.)

Для некоторых типов приложений, таких как приложения Universal Windows App (UWP), вы не прикрепляете непосредственно к названию процесса, а используете опцию меню **пакета приложений Debug Installed Package** в Visual Studio (см. таблицу).

Чтобы отладчик мог присоединиться к коду на языке C++, код должен предоставлять `DebuggableAttribute`. Это можно добавить в код автоматически, путем связывания с параметром [/ASSEMBLYDEBUG](/cpp/build/reference/assemblydebug-add-debuggableattribute) компоновщика.

Для отладки сценария на стороне клиента в браузере должна быть включена отладка скриптов. Для отладки сценария на стороне клиента в Chrome выберите **веб-кит** в качестве типа кода, и в зависимости от типа приложения, возможно, потребуется закрыть все экземпляры Chrome и запустить браузер в режиме отладки (тип `chrome.exe --remote-debugging-port=9222` из командной строки).

Чтобы быстро выбрать запущенный процесс, чтобы прикрепить к, в Visual Studio, **введите Ctrl**+**Alt**+**P**, а затем введите первую букву названия процесса.

## <a name="use-debugger-features"></a>Использование функций отладчика

Чтобы использовать все функции отладчика Visual Studio (например, при подключении к процессу, приложение должно точно соответствовать вашему локальному источнику и символам. То есть отладчик должен уметь загружать правильные [файлы символа (.pdb).](../debugger/specify-symbol-dot-pdb-and-source-files-in-the-visual-studio-debugger.md) По умолчанию для этого требуется сборка отладки.

Для сценариев удаленной отладки в Visual Studio уже должен быть открыт исходный код (или копия исходного кода). Собранные файлы аппликажа на удаленной машине должны поступать из той же сборки, что и на локальной машине.

В некоторых локальных сценариях отладки можно отладить Visual Studio без доступа к источнику, если в приложении присутствуют правильные файлы символов. По умолчанию для этого требуется сборка отладки. Для получения дополнительной информации [см.](../debugger/specify-symbol-dot-pdb-and-source-files-in-the-visual-studio-debugger.md)

## <a name="troubleshoot-attach-errors"></a><a name="BKMK_Troubleshoot_attach_errors"></a> Устранение ошибок присоединения
 При присоединении отладчика к выполняющемуся процессу этот процесс может содержать один или несколько типов кода. Типы кода, к которым может присоединиться отладчик, отображаются и выбираются в диалоговом окне **Выбор типа кода** .

 Иногда отладчик может успешно присоединяться к одному типу кода, но не к другому. Такое может происходить при попытке присоединения к процессу, выполняющемуся на удаленном компьютере. На удаленном компьютере для одних типов кода могут иметься компоненты удаленной отладки, а для других — нет. Такое также может происходить при попытке присоединиться к двум или более процессам для прямой отладки базы данных. Отладка SQL поддерживает присоединение только к одному процессу.

 Если отладчик может прикрепляться к некоторым, но не ко всем типам кода, вы видите сообщение, идентифицируя, какие типы не прикрепили.

 Если отладчик успешно подключается хотя бы к одному типу кода, то можно приступать к отладке процесса. Но отлаживать можно будет только те типы кода, к которым удалось подсоединиться. Неприкрепленный код в процессе будет по-прежнему выполняться, но вы не сможете устанавливать точки разрыва, просматривать данные или выполнять другие операции отладки в этом коде.

 Если требуется более конкретная информация о том, почему отладчик не прикрепится к типу кода, попробуйте прикрепить только этот тип кода.

 **Чтобы получить подробные сведения о том, почему не удалось присоединиться к типу кода:**

1. Отключитесь от процесса. В меню **Debug** выберите **Detach All**.

1. Прикрепите к процессу, выбрав только тип кода, который не удалось прикрепить.

    1. В диалоговом окне **Присоединение к процессу** выберите процесс в списке **Доступные процессы**.

    2. Щелкните **Выбрать**.

    3. В диалоговом окне **Выбор типа кода** выберите **Выполнять отладку кода следующих типов** и выберите тип кода, к которому не удалось присоединиться. Отменить другие типы кода.

    4. Нажмите кнопку **ОК**.

    5. В диалоговом поле **Attach to Process** выберите **Прикрепите**.

    На этот раз присоединение не пройдет полностью, и будет выдано сообщение о конкретной ошибке.
