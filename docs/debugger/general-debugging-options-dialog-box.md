# <a name="general-debugging-options"></a>Общие параметры отладки

Чтобы задать параметры отладчика Visual Studio, выберите **сервис**  > **Параметры**и в разделе **Отладка** установите или снимите флажки рядом с параметрами **Общие** . Вы можете восстановить все параметры по умолчанию с помощью **средств**  > **импорта и экспорта**  > **Сбросить все параметры**. Чтобы сбросить подмножество параметров, сохраните параметры с помощью **мастера импорта и экспорта параметров** перед внесением изменений, которые необходимо протестировать, а затем импортируйте сохраненные параметры позже.

Можно задать следующие **Общие** параметры.

**Запрашивать подтверждение перед удалением всех точек останова**: требует подтверждения перед завершением команды **удаления всех точек останова** .

**Прерывать все процессы при прерывании одного процесса**: одновременно прерывает работу всех процессов, к которым присоединен отладчик, при возникновении перерыва.

**Прерывать в случае исключений между AppDomain или управляемыми и собственными границами**: при отладке в управляемом или смешанном режиме среда CLR может перехватывать исключения, пересекающие границы домена приложений, или границы управляемого и машинного кода, если следующее условия имеют значение true:

1. Машинный код вызывает управляемый код с использованием COM-взаимодействия; при этом в управляемом коде возникает исключение. См. статью [Общие сведения о COM-взаимодействии](/dotnet/articles/visual-basic/programming-guide/com-interop/introduction-to-com-interop).

2. Управляемый код, выполняемый в домене приложения 1, вызывает управляемый код домена приложения 2; при этом в управляемом коде домена приложения 2 возникает исключение. См. [Программирование с использованием доменов приложений и сборок](/dotnet/articles/framework/app-domains/index).

3. Когда код вызывает функцию с помощью отражения, и эта функция создает исключение. См. раздел [отражение](/dotnet/framework/reflection-and-codedom/reflection).

В условиях 2 и 3 исключение иногда перехватывается управляемым кодом в `mscorlib`, а не средой CLR. Этот параметр не влияет на прерывание по исключениям, перехватываемым с помощью библиотеки `mscorlib`.

**Включить отладку на уровне адреса**: включает расширенные функции для отладки на уровне адреса (окно **дизассемблирования** , окно **регистры** и адреса точек останова).

- **Показывать дизассемблированный код, если источник недоступен**: автоматически отображает окно **дизассемблирования** при отладке кода, для которого исходный код недоступен.

**Включить фильтры точек останова**: позволяет устанавливать фильтры для точек останова, чтобы они повлияли только на определенные процессы, потоки или компьютеры.

**Использовать новый вспомогательный метод исключения**: включает вспомогательное приложение для исключений, заменяющее помощник по исключениям. (Вспомогательный метод исключения запускается в Visual Studio 2017)

> [!NOTE]
> Для управляемого кода этот параметр был ранее вызван **«Включение помощника по исключениям** ».

**Включить только мой код**: отладчик отображает и пошаговые инструкции в пользовательском коде ("мой код"), игнорируя системный код и другой код, оптимизированный или не имеющий отладочных символов.

- **Предупреждать при отсутствии пользовательского кода при запуске (только управляемый)** : при запуске отладки с только мой код включенным будет выдаваться предупреждение, если пользовательский код отсутствует ("мой код").

**Включить .NET Framework пошагового выполнения в источнике**: позволяет отладчику выполнять шаг с заходом в источник .NET Framework. Если включить этот параметр, Только мой код автоматически отключается. .NET Framework символы будут скачаны в расположение кэша. Измените расположение кэша, используя диалоговое окно **Параметры** , категория **Отладка** , страница **символы** .

**Шаг с обходом свойств и операторов (только управляемый)** : предотвращает пошаговое выполнение отладчика по переходу к свойствам и операторам в управляемом коде.

**Включить вычисление свойств и другие неявные вызовы функций**: включает автоматическое вычисление свойств и неявные вызовы функций в окнах переменных и диалоговом окне " **Быстрая проверка** ".

- **Функция преобразования строк вызова для объектов в окнах переменныхC# (только для JavaScript)** : выполняет вызов неявного преобразования строки при вычислении объектов в окнах переменных. Результат отображается в виде строки, а не имени типа. Применимо только при отладке кода C#. Этот параметр может быть переопределен атрибутом DebuggerDisplay (см. раздел [Использование атрибута DebuggerDisplay](../debugger/using-the-debuggerdisplay-attribute.md)).

**Включить поддержку исходного сервера**: сообщает отладчику Visual Studio о получении исходных файлов с исходных серверов, реализующих протокол SrcSrv (`srcsrv.dll`). Team Foundation Server и инструменты отладки для Windows — два исходных сервера, которые реализуют этот протокол. Дополнительные сведения о настройке SrcSrv см. в документации по [SRCSRV](/windows-hardware/drivers/debugger/srcsrv) . Кроме того, см. раздел [Указание символов (. pdb) и исходных файлов](../debugger/specify-symbol-dot-pdb-and-source-files-in-the-visual-studio-debugger.md).

> [!IMPORTANT]
> Так как при чтении *PDB*-файлов может выполняться произвольный код в файлах, убедитесь, что вы доверяете серверу.

- **Распечатывать сообщения диагностики исходного сервера в окне вывода**: когда включена поддержка исходного сервера, этот параметр включает диагностическое отображение.

- **Разрешить использовать исходный сервер для сборок с частичным доверием (только для управляемого кода)** : Если включена поддержка исходного сервера, этот параметр переопределяет поведение по умолчанию не извлекать источники для сборок с частичным доверием.

- **Всегда выполнять ненадежные команды исходного сервера без запроса**: Если включена поддержка исходного сервера, этот параметр переопределяет поведение по умолчанию при запросе непроверенной команды.

**Включить поддержку ссылки на исходный код**: указывает отладчику Visual Studio загрузить исходные файлы для *pdb* -файлов, содержащих сведения об исходной ссылке. Дополнительные сведения об исходной ссылке см. в [спецификации исходной ссылки](https://github.com/dotnet/core/blob/master/Documentation/diagnostics/source_link.md).

> [!IMPORTANT]
> Так как исходная ссылка будет скачивать файлы с помощью HTTP или HTTPS, убедитесь, что вы доверяете *pdb* -файлу.

- **Вернуться к проверке подлинности диспетчера учетных данных Git для всех исходных запросов ссылки**: Если включена поддержка ссылки на исходный код и запрос на исходную ссылку не проходит проверку подлинности, Visual Studio вызывает диспетчер учетных данных Git.

**Выделить всю строку для точек останова и текущего оператораC++ (только)** : когда отладчик выделит точку останова или текущую инструкцию, она выделяет всю строку.

**Требовать точного соответствия исходных файлов исходной версии**: указывает отладчику убедиться, что исходный файл соответствует версии исходного кода, используемой для отладки исполняемого файла. Если версия не совпадает, вам будет предложено найти соответствующий источник. Если соответствующий исходный код не найден, в процессе отладки исходный код не отображается.

**Перенаправлять весь текст окна вывода в окно интерпретации**: отправляет все сообщения отладчика, которые обычно отображаются в окне **вывода** , в окно **интерпретации** .

**Показывать необработанную структуру объектов в окнах переменных**: отключает все настройки представления структуры объектов. Дополнительные сведения о настройках представления см. в разделе [Создание настраиваемых представлений управляемых объектов](../debugger/create-custom-views-of-dot-managed-objects.md).

**Отключить JIT-оптимизацию для загрузки модуля (только управляемый код)** : отключает JIT-оптимизацию управляемого кода при загрузке модуля и компилирует JIT при присоединенном отладчике. Отключение оптимизации позволяет упростить процесс отладки некоторых проблем, однако может отрицательно сказаться на производительности. В режиме "Только мой код" при отключении JIT-оптимизации в коде пользователя ("Мой код") может отображаться посторонний код. Дополнительные сведения см. в разделе [JIT-оптимизация и отладка](../debugger/jit-optimization-and-debugging.md).

**Включить отладку JavaScript для ASP.NET (Chrome, ребр и IE)** : включает отладчик скриптов для приложений ASP.NET. При первом использовании в Chrome может потребоваться войти в браузер, чтобы включить установленные расширения Chrome. Отключите этот параметр, чтобы вернуться к старому поведению.

**Включение пограничных средства для разработчиков для приложений JavaScript UWP (экспериментальных)** : включает средства разработчика для приложений JavaScript UWP в Microsoft ребро.

**Включить устаревший отладчик JavaScript Chrome для ASP.NET**: включает устаревший отладчик скриптов JavaScript Chrome для приложений ASP.NET. При первом использовании в Chrome может потребоваться войти в браузер, чтобы включить установленные расширения Chrome.

**Используйте экспериментальный способ для запуска отладки JavaScript Chrome при запуске Visual Studio от имени администратора**: сообщает Visual Studio о попыток запуска Chrome при отладке JavaScript.

**Загружать экспорты DLL (только машинный код)** : загружает таблицы экспорта DLL. Символьные данные из таблиц экспорта библиотеки DLL можно использовать при работе с сообщениями Windows, процедурами Windows (WindowProcs), COM-объектами, при маршалинге или при работе с любой библиотекой DLL, для которой нет символов. Считывание данных экспорта библиотеки DLL связано с определенными дополнительными затратами. Поэтому данная возможность по умолчанию отключена.

Чтобы посмотреть, какие символы доступны в таблице экспорта библиотеки DLL, воспользуйтесь командой `dumpbin /exports`. Символы доступны для любой 32-разрядной системной библиотеки DLL. В выходных данных команды `dumpbin /exports` можно увидеть точное имя функции, включая символы, отличные от буквенно-цифровых. Это полезно при задании точки останова в функции. Имена функций из таблиц экспорта библиотеки DLL могут отображаться в отладчике в сокращенном виде. Вызовы функций перечисляются в том порядке, в котором эти функции вызываются, при этом текущая функция (наиболее глубоко вложенная) располагается наверху. Дополнительные сведения см. в разделе [dumpbin /exports](/cpp/build/reference/dash-exports).

**Показать диаграмму параллельных стеков снизу вверх**: управляет направлением, в котором стеки отображаются в окне **Параллельные стеки** .

**Игнорировать исключения доступа к памяти GPU, если записанные данные не изменили значение**: игнорирует состояния гонки, обнаруженные во время отладки, если данные не были изменены. Дополнительные сведения см. в разделе [Отладка кода GPU](../debugger/debugging-gpu-code.md).

**Использовать режим совместимости управляемого**кода: заменяет модуль отладки по умолчанию устаревшей версией для включения следующих сценариев:

- Вы используете .NET Framework язык, отличный от C#, Visual Basic или F# предоставляющий собственный вычислитель выражений (включая C++/CLI).

- При отладке в смешанном режиме необходимо C++ включить режим "изменить и продолжить" для проектов.

> [!NOTE]
> Выбор режима совместимости управляемого кода отключает некоторые функции, реализованные только в модуле отладки по умолчанию. Модуль отладки прежних версий был заменен в Visual Studio 2012.

**Использование вычислителей выражений прежних версий C# и VB**. отладчик будет использовать Visual Studio 2013 C# или Visual Basic вычислители выражений, а не вычислители выражений на основе Visual Studio 2015 Roslyn.

**Предупреждать при использовании настраиваемых визуализаторов отладчика для потенциально небезопасных процессов (только управляемые)** : Visual Studio предупреждает о том, что вы используете Пользовательский визуализатор отладчика, выполняющий код в отлаживаемом процессе, так как это может привести к запуску небезопасного кода .

**Включить распределитель кучи отладки Windows (только машинный код)** : включает отладочную кучу Windows для улучшения диагностики кучи. Включение этого параметра повлияет на производительность отладки.

**Включить средства отладки пользовательского интерфейса для XAML**: динамическое визуальное дерево и интерактивное проводник для свойств при запуске отладки (**F5**) поддерживаемый тип проекта. Дополнительные сведения см. в разделе [Проверка свойств XAML во время отладки](../xaml-tools/inspect-xaml-properties-while-debugging.md).

- **Предварительный просмотр выбранных элементов в динамическом визуальном дереве**: элемент XAML, контекст которого выбран, также выбирается в **динамическом окне визуального дерева** .

- **Показать средства среды выполнения в приложении**: отображает команды **динамического визуального дерева** на панели инструментов в главном окне отлаживаемого приложения XAML. Этот параметр появился в Visual Studio 2015 с обновлением 2.

- **Включить горячую перезагрузку XAML**: позволяет использовать функцию горячей загрузки XAML с кодом XAML, когда приложение выполняется. (Эта функция ранее называлась "XAML Edit и Continue").

**Включить средства диагностики во время отладки**: во время отладки появится окно **средства диагностики** .

**Показать затраченное время подсказку с затраченным во время отладки**: в окне кода отображается время, прошедшее с момента выполнения данного вызова метода при отладке.

**Включить "изменить и продолжить**": включает функцию "изменить и продолжить" во время отладки.

- **Включение собственного режима "изменить и продолжить**". можно использовать функции "изменить и продолжить" C++ во время отладки машинного кода. Дополнительные сведения см. в разделе ["изменить иC++продолжить" ()](../debugger/edit-and-continue-visual-cpp.md).

- **Применить изменения к Continue (только машинный код)** : Visual Studio автоматически компилирует и применяет все невыполненные изменения кода, сделанные при продолжении процесса из состояния останова. Если этот параметр не выбран, можно применить изменения с помощью пункта **Применить изменения кода** в меню **Отладка**.

- **Предупреждать об устаревшем коде (только машинный код)** : получение предупреждений об устаревшем коде.

**Показать запуск для нажатия кнопки в редакторе во время отладки**: Если выбран этот параметр, при отладке будет отображаться кнопка [выполнить до нажатия](../debugger/debugger-feature-tour.md#run-to-a-point-in-your-code-quickly-using-the-mouse) .

**Автоматически закрывать консоль при остановке отладки**: указывает Visual Studio закрыть консоль в конце сеанса отладки.

::: moniker range=">= vs-2019"
**Включить быстрое вычисление выражений (только управляемый)** : позволяет отладчику быстрее выполнять оценку, имитируя выполнение простых свойств и методов.
::: moniker-end

## <a name="options-available-in-older-versions-of-visual-studio"></a>Параметры, доступные в более ранних версиях Visual Studio

Если вы используете более раннюю версию Visual Studio, могут присутствовать некоторые дополнительные параметры.

**Включить помощник по исключениям**: для управляемого кода включает помощник по исключениям. Начиная с Visual Studio 2017, в помощнике по исключениям заменяется обработчик исключений.

**Очистка стека вызовов от необработанных исключений**: заставляет окно **стека вызовов** выполнить откат стека вызовов до точки, прежде чем возникло необработанное исключение.

**Предупреждать при отсутствии символов при запуске (только машинный код)** : отображает диалоговое окно предупреждения при отладке программы, для которой отладчик не имеет символьной информации.

**Предупреждать при отключении отладки скриптов при запуске**: отображает диалоговое окно с предупреждением при запуске отладчика с отключенной отладкой скриптов.

**Использовать режим совместимости машинного кода**. Если выбран этот параметр, отладчик использует собственный отладчик Visual Studio 2010 вместо нового отладчика машинного кода.

- Используйте этот параметр при отладке кода .NET C++ , так как новый модуль отладки не поддерживает вычисление выражений .NET C++ . Однако включение режима совместимости машинного кода отключает множество функций, которые зависят от работы текущей реализации отладчика. Например, в устаревшем механизме не хватает многих визуализаторов для встроенных типов, таких как `std::string` в проектах Visual Studio 2015.   Используйте Visual Studio 2013 проекты для оптимального процесса отладки в таких случаях.
