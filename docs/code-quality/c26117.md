# <a name="c26117"></a>C26117
C26117 предупреждения: Освобождение неудерживаемой блокировки \<lock > в функции @no__t — 1func >.

 Применение *пар блокировки* с синтаксическим *ограничением и блокировок* в C/C++ Program не выполняется языком. Функция может вызвать побочный результат блокировки, делая наблюдаемое изменение в состоянии параллелизма. Например, функция-оболочка блокировки увеличивает количество операций получения блокировки или число блокировок для данной блокировки. Можно добавить аннотацию к функции, которая имеет побочный результат из запроса блокировки или снятия блокировки с помощью `_Acquires_lock_` или `_Releases_lock_` соответственно. Без таких заметок функция должна не изменять счетчик блокировок после возврата. Если получение и выпуски не сбалансированы, они считаются *потерянными*. Предупреждение C26117 выдается, когда функция, которая не помечена с помощью `_Releases_lock_`, освобождает блокировку, которая не удерживается, так как функция должна владеть блокировкой, прежде чем она будет освобождена.

## <a name="example"></a>Пример
 В следующем примере создается предупреждение C26117, так как функция, `ReleaseUnheldLock`, освобождает блокировку, которая не обязательно удерживается — состояние `flag` является неоднозначным, и нет аннотации, указывающей, что она должна.

```cpp

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

int flag;

void ReleaseUnheldLock(DATA* p)
{
    if (flag)
        EnterCriticalSection(&p->cs);
    // code ...
    LeaveCriticalSection(&p->cs);
}
```

## <a name="example"></a>Пример
 Следующий код устраняет проблему, гарантируя, что освобожденная Блокировка также будет получена в тех же условиях.

```cpp

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

int flag;

void ReleaseUnheldLock(DATA* p)
{
    if (flag)
    {
        EnterCriticalSection(&p->cs);
        // code ...
        LeaveCriticalSection(&p->cs);
    }
}
```
